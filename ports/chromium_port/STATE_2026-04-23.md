Honest accounting of what's shipped, what's plumbed-but-untested,

## Shipped (in-kernel plumbing)

- `src/ui/shell.rs::cmd_chromium` — `chromium [flags] <url>` shell
  command, builds argv, default flags (`--headless --no-sandbox
  --disable-gpu --single-process --ozone-platform=headless
  --window-size=1280x1024`), dispatches to `runner::run_chromium`.
- `src/batcave/linux/runner.rs::run_chromium` — locates the blob via
  `initrd::locate_chromium_blob()`, verifies CRC32 and (if
  `INITRD_PUBKEY` is set) Ed25519 signature, refuses to load an
  unsigned blob unless `ALLOW_UNSIGNED_INITRD = true`, allocates a
  cave slot, loads the ELF rebased at `CHROMIUM_VIRT_BASE =
  0x10000000`, installs a dedicated per-cave L1 page table, and
  hands execution to the BatCave runner.
- `src/kernel/mm/initrd.rs` — walks the `[BATCHROM][size
  u64][blob][crc32][CHROMEND]` framing starting at `__kernel_end`,
  up to a safety cap past the kernel image.
- `src/drivers/display/chromium_blit.rs` — receives Chromium's pixel
  buffer via a mapped page at `/batos/fb0` and blits to the
  framebuffer. Wired up at boot.
- `tools/bake_chromium.sh` — concatenates a content_shell ELF onto
  the kernel image with the expected framing.
- `tools/run_chromium.sh` — boots QEMU with 4 GB RAM and the baked
  kernel.
- `scripts/qemu_chromium_pipeline_smoke.py` — smoke test that bakes
  a stand-in ELF (`tests/hello`) and invokes `chromium
  https://example.com`. Used to reveal the gap documented below.
- All 7 GN-arg / Dockerfile / build.sh prep files in
  `ports/chromium_port/`.

## Real gap — the initrd blob never reaches memory today

When `tools/bake_chromium.sh` runs, it concatenates the BATCHROM
framing onto the kernel ELF on disk:

```
[kernel ELF PT_LOADs] [BATCHROM][size][content_shell][crc32][CHROMEND]
```

QEMU's `-kernel` path, when given an ELF, loads *only the PT_LOAD
segments* into RAM. The appended blob bytes sit in the file but
never get copied to memory. As a result
`initrd::locate_chromium_blob()` reads zero-filled memory past
`__kernel_end` and returns `None` every time — the shell reports
"error: no Chromium binary baked into this image" even when the
bake succeeded.

Verified via `qemu_chromium_pipeline_smoke.py` today:

```
✓ initrd:blob locate         (code path reached)
✓ shell cmd accepted         (`chromium https://example.com` parsed)
✓ runner reached             (cmd_chromium dispatched)
✗ initrd sig check           (never got past "no Chromium binary")
```

This is a framing-layer bug, not a Chromium bug. Three candidate
fixes:

1. **Flat-binary bake.** `objcopy -O binary bat_os bat_os.bin`, then
   append. But the flat-binary's file size is just `.text + .rodata
   + .data` whereas `__kernel_end` is after `.bss + stack` — so the
   blob would land INSIDE bss/stack. Need to either move
   bss/stack earlier in the link order, or point the locator at
   `file_size_of_flat_bin` instead of `__kernel_end`. Lowest risk
   if we also add a linker symbol for the file-end anchor.

2. **QEMU `-initrd`.** Use QEMU's initrd loading (it puts the blob
   at a known physical address and fills device-tree / boot-params
   with its location). Kernel reads that instead of scanning past
   `__kernel_end`. Cleanest path but requires `src/main.rs` boot
   entry to pick up boot-params (we currently ignore the dtb/x0
   argument passed in by QEMU).

3. **Custom PT_LOAD segment.** Add a `.initrd` section in
   `linker.ld` that's covered by a PT_LOAD but sized at link time
   only by a placeholder. Then `tools/bake_chromium.sh` patches the
   placeholder + program-header size. Invasive linker work; also
   inflates the shipped kernel file even when no blob is baked.

Recommendation: **option 2**. It matches how real bootloaders work
on aarch64 (m1n1 on M4 already passes boot-args; QEMU emulates that
contract for `-kernel + -initrd`). Much less risk to the existing
linker story, and removes the "append to kernel" dance entirely.

## What we landed today (partial progress on option 2)

This session attempted option 2 and proved it half-works — enough
to point the next session at what's left:

- `src/boot/dtb.rs` learned `/chosen/linux,initrd-start` +
  `linux,initrd-end` (accepts u32 or u64 per FDT convention). Now
  exposed as `DtbInfo.initrd_start` / `initrd_end`.
- `src/kernel/mm/initrd.rs::set_range(start, end)` lets the boot
  code override the probe base. `probe()` prefers the override
  when set; falls back to `__kernel_end` scan otherwise.
- `src/main.rs` wires DTB-parsed initrd range into `initrd::set_range`
  when present.
- `tools/bake_chromium_initrd.sh` produces a standalone BATCHROM-
  framed blob suitable for QEMU `-initrd`.
- `src/arch/aarch64/boot.s` preserves x0 (QEMU's DTB pointer)
  across the mpidr check — wasn't needed for QEMU (which uses
  linux_header.s) but fixes the historical bug in the unused boot
  stub.

### The catch we hit

QEMU's `-kernel <ELF>` path does **not** honour the ARM64 Linux
boot protocol — x0 is zeroed, not set to the DTB. The DTB IS
generated by QEMU (`-machine virt` always synthesises one), but
it's only passed when the kernel is a **flat `Image`-format
binary**, not when it's an ELF. Confirmed via a temporary debug
print of `dtb_ptr` from `kernel_main`: always `0x0` under the
current `-kernel bat_os -initrd ...` invocation.

So the DTB path is correct kernel-side and will fire the first
time we boot under a bootloader that actually delivers the DTB
(m1n1 on real M4 already does this; QEMU wants a different kernel
packaging).

### To unblock Chromium delivery

Pick one:

a. **Objcopy the kernel to a flat `Image`.** `aarch64-linux-gnu-
   objcopy -O binary bat_os bat_os.bin`, then boot with `-kernel
   bat_os.bin -initrd chromium_initrd.bin`. QEMU will honour the
   Linux boot protocol → x0 = DTB → our code works.
   Risk: `__kernel_end` no longer in a reliable place for the
   legacy appended-blob fallback, but the DTB path doesn't need it.

b. **Use virtio-blk as the transport.** Write the framed blob to
   a raw disk image, attach as `-drive ... -device virtio-blk-
   device`, teach `initrd::probe()` to read sectors via
   `drivers::virtio::blk::read_sectors`. Bonus: exercises the
   virtio-blk driver we just shipped. Downside: reading 150 MB one
   sector at a time is slow; would want DMA.

c. **Hook the `-fw_cfg` mechanism.** QEMU-specific but delivers
   blobs at a known path regardless of kernel format. Would need
   a fw_cfg client in the kernel.

Recommendation for next session: **option (a)**. Single-line
addition to the build + runner scripts, no new kernel code.

## Still-open items (Phase 2 onwards)

Unchanged from `CHROMIUM_PORT_PLAN.md`:

- **Phase 0-1 build**: Dockerfile + .gn-args + build.sh are ready.
  Nobody's run the full `fetch chromium` + `gn gen` + `autoninja
  content_shell` yet. That's a 4-8 hour build on M-series once
  kicked off.
- **Phase 2 static linking**: musl sysroot, `-static-pie`,
  BoringSSL in the binary, etc. Days of iteration.
- **Phase 3 syscall census**: strace under `content_shell --version`
  + `--screenshot=out.png https://example.com`, categorize the
  ~150-200 distinct syscalls, update `SYSCALL_RESEARCH.md`.
- **Phase 4 syscall implementation**: futex, clone, epoll, eventfd,
  timerfd, getrandom, signalfd, real threading. 1-2 weeks of
  kernel work — the biggest single chunk.
- **Phase 5 display backend**: Ozone-batos or patched headless
  that hands pixels to `chromium_blit`.
- **Phase 6 network bridge**: socket/connect/send/recv/epoll
  glue from the BatCave runner to `src/net/tcp.rs`.
- **Phase 7 boot integration**: depends on the framing fix above.
