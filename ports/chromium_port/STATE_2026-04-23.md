# Chromium port — state as of 2026-04-23 (evening v2)

## ✅ Real content_shell + real glibc: in-kernel dynamic linking works

Added a multi-file initrd archive format + in-kernel dynamic linker.
Baked content_shell (293 MB) plus 12 of its DT_NEEDED shared libraries
(libc.so.6, libm.so.6, libpthread.so.0, ld-linux-aarch64.so.1, libdl,
libexpat, libgcc_s, libnspr4, libnss3, libnssutil3, libplc4, libplds4 —
all pulled from a real debian:bookworm arm64 image, not the sysroot
stubs) into one 284 MB initrd blob. The kernel now:

1. Loads all 13 ELFs into one contiguous 188 MB cave slab.
2. Walks each lib's .dynamic / .rela.dyn / .rela.plt.
3. Does a cross-module symbol-resolution pass: for every UNDEF
   GLOB_DAT / JUMP_SLOT in content_shell, search DT_SYMTAB of all
   loaded libs by name; if a DEFINED symbol matches, write the real
   cave-VA into the GOT slot. Unresolved entries fall back to the
   0xBAD0… sentinel.

Latest smoke test:

So content_shell now successfully calls `__libc_start_main`, runs
inside the real glibc, does whatever it does, and eventually reaches
a function pointer containing uninitialized data (TLS slot or
something else glibc expects the dynamic linker to have set up).
This is the exact "glibc needs init" boundary the plan predicted.

## Critical bug fixed along the way

With a 284 MB initrd loaded at physical 0x48000000 (QEMU `-initrd`),
the frame allocator's "pool" covered `[past_heap, MEMORY_END)` but
DIDN'T reserve the initrd bytes. The first big multi-ELF phys
allocation (188 MB) came back at 0x5AC00000 — **inside** the initrd
region. content_shell's PT_LOAD copy then smashed the baked-in
libraries out from under us ("0 REL" for every lib). Fix:

- `initrd::blob_phys_range()` exposes the real `(start, end)` of the
  delivered blob.
- `mm::init` calls `frame::reserve_range(ir_start, ir_end)` right
  after `frame::init` so the allocator treats the initrd bytes as
  in-use from the start.

## Still-open, in rough priority

1. **DT_INIT_ARRAY execution.** Every lib has its init constructors
   that glibc-based dynamic linking would run before handing control
   to the main exe's entry. Without them, libc's own globals (malloc
   tables, stdio FILE objects, locale info, mutex state) are zero
   and the first call that depends on them fails.
2. **TLS setup.** glibc uses `tpidr_el0` as the TLS pointer. Each
   thread (including the main one) needs a TLS block allocated and
   `tpidr_el0` set to it. All `errno` / `stdout` / `__environ`
   accesses go through TLS.
3. **DT_VERSYM / DT_VERNEED versioned symbol lookup.** We do plain
   name match. Real glibc symbols are versioned (`printf@GLIBC_2.17`
   vs `@GLIBC_2.34`). Our match picks the first hit, which might be
   the wrong version and hit an ABI difference.
4. **The remaining 25 UNDEFs in content_shell** — compiler-rt helpers,
   symbols from libraries we didn't pack. Dump them with a sentinel-
   to-name resolver and either pack more libs or implement stubs.
5. **IRELATIVE resolvers.** We write the resolver's address as a
   survivable approximation; actually running the resolver at load
   time (in EL1 with the cave's page table active) is next.

None of these are "all or nothing" — progress is measurable at each
step, the smoke test gives a concrete diagnostic, and each fix
either reduces UNDEF counts or pushes the crash deeper.

## ✅ Real `content_shell` now loads + reaches `__libc_start_main`

`scripts/qemu_chromium_pipeline_smoke.py` walks a **real 280 MB
content_shell** (pre-built from a previous iteration, sitting in the
`batos-chromium-src` Docker volume at `src/out/BatOs/content_shell`)
all the way through to the first libc call:

```
[loader] Rebased Entry: 0x0000000001720000 virt_base: 0x0000000010000000
[loader] Rebased phys_base: 0x0000000055c00000
[loader] Applied 539446 REL, 50 GLOB, 550 JUMP, 5 IREL (rebased)
[loader] --- executing ---
!!! UNHANDLED SYNC EXCEPTION !!!
  EC: 0x0000000000000020                ← Instruction abort (lower EL)
  ELR: 0xbad0000000000020               ← Sentinel: undef sym #2
[smoke] UNDEF symbol call: sym #2 = __libc_start_main
```

So `content_shell` went from "won't even load" to "runs its C runtime
startup, finishes applying 540k relocations at EL0, and calls
`__libc_start_main`" — exactly the Phase 2 / Phase 4 boundary the
plan predicted.

What had to land to get here (all in one commit):

- **Widened the cave user window** from 100 × 2 MB (200 MB) to
  200 × 2 MB (400 MB) via new `mmu::CAVE_BLOCKS`. `content_shell`
  is ~162 MB of PT_LOAD + 1 MB stack, so 200 MB wasn't quite enough.
- **Extended kernel identity map to a full 2 GB** (L1[2] → L2_xhi
  covering 0x80000000..0xBFFFFFFF). Previous 256 MB window broke
  two flows: kernel-pool frames at ~0xBFFFX000 (written by cave
  page-table setup) and ELF phys_base + 162 MB landing past
  0x50000000. DATA ABORT DFSC=0x06 fixed.
- **Bumped `initrd::MAX_BLOB_SIZE`** from 256 MB to 512 MB —
  the 280 MB framed initrd was rejected as "implausible."
- **Loader now applies GLOB_DAT, JUMP_SLOT, and IRELATIVE** on
  top of R_RELATIVE. Walks DT_SYMTAB for the symbol's vaddr and
  DT_JMPREL for the separate PLT rela table (340 entries in
  content_shell, all of which were being silently ignored).
- **Undef-symbol sentinel**: when a GLOB_DAT / JUMP_SLOT's symbol
  is SHN_UNDEF (glibc functions like `__libc_start_main`,
  `printf`, `pthread_*`, compiler-rt helpers — 567 of them in
  today's binary), the loader writes
  `0xBAD0_0000_0000_0000 | (sym_idx << 4)` into the GOT slot.
  When EL0 calls through, we get a clean instruction abort with
  the ELR still decodable. The smoke script reads ELR, extracts
  the symbol index, and resolves the name via DT_STRTAB.
- **`ports/chromium_port/out/content_shell`** checked out of the
  Docker volume (293 MB file; ~162 MB of PT_LOAD memory image).

## Known follow-up: 567 undefined libc symbols

`content_shell` as currently built is **dynamically linked against
glibc** (the Chromium default sysroot). The PLT holds 340 stubs
expecting lazy resolution by a real dynamic linker, and the symbol
table has 567 SHN_UNDEF entries ranging from `abort`,
`__libc_start_main`, `__cxa_finalize`, stdio (`printf`, `fwrite`,
`stdin`/`stdout`/`stderr`), full pthread surface
(`pthread_mutex_*`, `pthread_cond_*`, `pthread_key_*`,
`pthread_sigmask`), compiler-rt arithmetic helpers
(`__divti3`, `__modti3`, `__floattidf`, `__powidf2`, …), libc
syscall wrappers (`open64`, `read`, `getrandom`, `gettid`,
`copy_file_range`, `statx`), and Chromium's own cross-module
exports from `base` and `v8`.

Two real ways forward (mutually exclusive):

**A. Static-libc rebuild** (cleanest). Update
`ports/chromium_port/.gn-args` + `Dockerfile` to link against musl
instead of the Chromium-bundled glibc sysroot. That's ~4–8 hours of
Docker build + GN-arg iteration. Requires ~60 GB free on the build
host — current Mac has only ~30 GB free, 29 GB in OrbStack's volume.
We'd need to either prune old images (~21 GB of 8 stale
`batos-chromium-build:progress-*` tags reclaimable) or offload to
the Ubuntu dev host.

**B. Kernel-side libc stubs.** Implement the 567 undef symbols as
Bat_OS syscall wrappers. For symbols that map cleanly to existing
syscalls (`read`, `write`, `getrandom`, `exit`) this is a few hours.
Compiler-rt arithmetic (~50 symbols) is almost trivial (copy libc++abi
implementations). Pthread is the big one — requires real futex,
clone, signal delivery, thread-local storage. Estimate: 2–4 weeks
for a full subset that gets to the display path.

`STATE_2026-04-23.md` originally recommended **A** (musl rebuild);
that's still the right answer. Stay on (A) even though the existing
Docker volume already contains a dynamically-linked build —
rebuilding with musl is far less work than reimplementing 567 libc
entry points.

What landed to make this work:

- **`load_elf_rebased` now allocates its stack** and sets the loader
  globals (`LOADED_STACK_PHYS`, `LOADED_PHYS_BASE`, `LOADED_ORIG_ENTRY`,
  and a new `LOADED_USER_VA_BASE`). Previously `execute_with_args`
  aborted with "no stack reserved" because the rebased path never
  touched those statics.
- **`execute_with_args` honours `LOADED_USER_VA_BASE`** in `to_uva`,
  in `stack_va_base`, in the final `user_sp` calculation, and in the
  entry-point selection (don't swap in `orig_entry` when user_va_base
  is non-zero — the caller's `info.virt_entry` is already the
  cave-VA-correct target).
- **Runner primes the MMU** via `setup_and_enable(info.phys_base)`
  before `switch_to_cave(l1)`. Chromium is often the first user
  binary of a session, so nobody else has turned the MMU on; the
  cave path assumes it's already up. The follow-up
  `setup_and_enable` inside `execute_with_args` is then a no-op
  (SCTLR.M=1), preserving the cave's L1.
- **Kernel-RAM mapping widened past `__text_end`** in both
  `setup_and_enable` and `setup_cave_pagetable_at`. Rust is
  currently scattering executable code into the rodata PT_LOAD
  (observed PC=0x402986ec after `msr sctlr_el1`, which is inside
  vaddr range 0x40200000..0x406E8A00 = rodata segment). With the
  original W^X split, block 1 got PXN and the first instruction
  fetch after MMU-enable aborted. Transitional fix: blocks past
  `__text_end` are now EL1-RW + EL1-exec + UXN. `linker.ld` also
  picks up `.text.cold.*` / `.text.unlikely.*` / `.text.hot.*` /
  `.text.startup.*` explicitly; once a toolchain bump actually
  gathers cold-path code into the .text PT_LOAD we can revert the
  widen to restore kernel W^X. Tracked as V9.
- **`BAT_OS_ALLOW_UNSIGNED_INITRD` env flag** gates the unsigned
  blob path in `runner::run_chromium`. Plumbed via `option_env!`
  with a `build.rs` rerun-if-env-changed hint so flipping it
  doesn't require `cargo clean`. Default is still REFUSE.
- **`initrd::probe` off-by-one fix**: changed `head + 16 >=
  ceiling` to `head + 16 > ceiling` (and likewise for the tail
  magic) — the old form refused the legitimate blob when it sat
  exactly against the ceiling.
- **`scripts/qemu_chromium_pipeline_smoke.py`** now objcopies the
  kernel to a flat `bat_os.bin` and boots `-kernel <flat> -initrd
  <blob>` so QEMU actually honours the ARM64 Linux boot protocol
  and hands the DTB in x0.

## Known follow-up: netsurf-THEN-chromium faults

If a primary-cave ELF (busybox/netsurf/freetype/…) runs first, the
MMU comes up with `PRIMARY_L1`, whose L2_high identity-maps only
0x40000000..0x50000000. But `alloc_kernel_frame()` hands cave
page-table pages from the top of memory (observed: 0xbfffe000).
When the next command is `chromium`, `setup_cave_pagetable_at`
tries to zero its newly allocated L1/L2 frames via the kernel's
identity view, but 0xbfffX000 isn't mapped in PRIMARY_L1 and we
get `DATA ABORT DFSC=0x05` (translation fault L1). Chromium-first
works because MMU is off during cave setup and writes go direct.

Fix: widen the kernel-RAM identity map to cover the whole
kernel-pool top-of-RAM range (or a permanent "all RAM" L2 mapped
at cave-setup time). Not on the critical path for delivery
verification; filing as a follow-up.

## Real gap — the initrd blob never reaches memory today

When `tools/bake_chromium.sh` runs, it concatenates the BATCHROM
framing onto the kernel ELF on disk:

```
[kernel ELF PT_LOADs] [BATCHROM][size][content_shell][crc32][CHROMEND]
```

QEMU's `-kernel` path, when given an ELF, loads *only the PT_LOAD
segments* into RAM. The appended blob bytes sit in the file but
never get copied to memory. As a result
`initrd::locate_chromium_blob()` reads zero-filled memory past
`__kernel_end` and returns `None` every time — the shell reports
"error: no Chromium binary baked into this image" even when the
bake succeeded.

Verified via `qemu_chromium_pipeline_smoke.py` today:

```
✓ initrd:blob locate         (code path reached)
✓ shell cmd accepted         (`chromium https://example.com` parsed)
✓ runner reached             (cmd_chromium dispatched)
✗ initrd sig check           (never got past "no Chromium binary")
```

This is a framing-layer bug, not a Chromium bug. Three candidate
fixes:

1. **Flat-binary bake.** `objcopy -O binary bat_os bat_os.bin`, then
   append. But the flat-binary's file size is just `.text + .rodata
   + .data` whereas `__kernel_end` is after `.bss + stack` — so the
   blob would land INSIDE bss/stack. Need to either move
   bss/stack earlier in the link order, or point the locator at
   `file_size_of_flat_bin` instead of `__kernel_end`. Lowest risk
   if we also add a linker symbol for the file-end anchor.

2. **QEMU `-initrd`.** Use QEMU's initrd loading (it puts the blob
   at a known physical address and fills device-tree / boot-params
   with its location). Kernel reads that instead of scanning past
   `__kernel_end`. Cleanest path but requires `src/main.rs` boot
   entry to pick up boot-params (we currently ignore the dtb/x0
   argument passed in by QEMU).

3. **Custom PT_LOAD segment.** Add a `.initrd` section in
   `linker.ld` that's covered by a PT_LOAD but sized at link time
   only by a placeholder. Then `tools/bake_chromium.sh` patches the
   placeholder + program-header size. Invasive linker work; also
   inflates the shipped kernel file even when no blob is baked.

Recommendation: **option 2**. It matches how real bootloaders work
on aarch64 (m1n1 on M4 already passes boot-args; QEMU emulates that
contract for `-kernel + -initrd`). Much less risk to the existing
linker story, and removes the "append to kernel" dance entirely.

to point the next session at what's left:

### The catch we hit

QEMU's `-kernel <ELF>` path does **not** honour the ARM64 Linux
boot protocol — x0 is zeroed, not set to the DTB. The DTB IS
generated by QEMU (`-machine virt` always synthesises one), but
it's only passed when the kernel is a **flat `Image`-format
binary**, not when it's an ELF. Confirmed via a temporary debug
print of `dtb_ptr` from `kernel_main`: always `0x0` under the
current `-kernel bat_os -initrd ...` invocation.

So the DTB path is correct kernel-side and will fire the first
time we boot under a bootloader that actually delivers the DTB
(m1n1 on real M4 already does this; QEMU wants a different kernel
packaging).

### To unblock Chromium delivery

Pick one:

a. **Objcopy the kernel to a flat `Image`.** `aarch64-linux-gnu-
   objcopy -O binary bat_os bat_os.bin`, then boot with `-kernel
   bat_os.bin -initrd chromium_initrd.bin`. QEMU will honour the
   Linux boot protocol → x0 = DTB → our code works.
   Risk: `__kernel_end` no longer in a reliable place for the
   legacy appended-blob fallback, but the DTB path doesn't need it.

b. **Use virtio-blk as the transport.** Write the framed blob to
   a raw disk image, attach as `-drive ... -device virtio-blk-
   device`, teach `initrd::probe()` to read sectors via
   `drivers::virtio::blk::read_sectors`. Bonus: exercises the
   virtio-blk driver we just shipped. Downside: reading 150 MB one
   sector at a time is slow; would want DMA.

c. **Hook the `-fw_cfg` mechanism.** QEMU-specific but delivers
   blobs at a known path regardless of kernel format. Would need
   a fw_cfg client in the kernel.

Recommendation for next session: **option (a)**. Single-line
addition to the build + runner scripts, no new kernel code.
