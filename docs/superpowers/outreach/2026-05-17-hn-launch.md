**URL field:** `https://sphragis.com` (or actual marketing-site URL — Kaden to confirm) **OR** `https://github.com/kadenlee1107/Sphragis` (probably stronger for an HN audience — the source is what they want to see, not the marketing page).

**Text field (first comment from submitter, posted seconds after submission):**

```
Hi HN — I'm the author. Sphragis is a memory-safe Rust microkernel
I've been building for ~14 weeks. It boots on real Apple M4 hardware
via an independent reverse-engineering pipeline (Asahi Linux doesn't
yet support M4 as of May 2026, and the M4/M5 RE work is still in
progress in the Asahi community).

Quick facts:
- ~96K LoC of no_std Rust, edition 2024
- Boots on M4 MacBook Pro 14" (Mac16,1 / J604 / T8132 "Donan") via
  m1n1 chainload from macOS Recovery
- TLS 1.3 with hybrid post-quantum key agreement (X25519+ML-KEM-768
  per draft-ietf-tls-ecdhe-mlkem-04), kernel-mediated — processes get
  plaintext fds and write HTTP, the kernel handles handshake + cert
  chain validation. No process can ship broken TLS or downgrade
  to HTTP.
- SealFS: AES-256-GCM-SIV (misuse-resistant AEAD), Argon2id master
  key derivation (8 MiB / 3 passes)
- Per-process default-deny network egress with audit-logged denials
- Driver coverage: dockchannel UART, AIC, PMGR, ATC PHY, DCP
  framebuffer, DWC3 XHCI USB
- 0 warnings + 0 clippy lints on pinned nightly
- Apache-2.0 (relicensed from AGPL on 2026-05-16)
- Source: https://github.com/kadenlee1107/Sphragis

What it deliberately doesn't have:
- No web browser. Browsing belongs on a separate device. What
  remains is the work that needs strong isolation — drafting,
  review, encrypted storage, signed communications.
- No package manager / app store. A small set of apps ship in the
  kernel image.
- No third-party kernel code. RustCrypto for crypto primitives is
  the only audited external dependency in the security path.

What's the point: I wanted to see how small a memory-safe
workstation OS could be if I picked a single hardware target and
deleted everything that wasn't load-bearing for "do focused work on
sensitive material on a laptop you trust." Apple Silicon M4 was a
pragmatic choice (the hardware is good, widely available, and there
was a gap in non-Apple OS support). Bare-metal Rust on M4 turned out
to be tractable as a single-person project on a 14-week horizon —
the boot-evidence photos on the site are from April 17, 2026.

Happy to answer technical questions about the boot path, the TLS
implementation, SealFS, or the M4 RE work. The marketing page has
the boot photos and full specifications; the source has the truth.

(Caveat: NOT yet FIPS certified, NOT yet on x86_64, the Verus proof
work on the cap dispatcher + IPC paths is partial-spec rather than
complete. Calling it production-ready would be dishonest; calling
it boot-on-real-hardware-with-real-properties is accurate.)
```
