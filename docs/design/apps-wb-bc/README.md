# Sphragis Wave 4 — WB · BC design source

Final wave of app redesigns. Generated by Claude Design (April
2026) in the same session as everything before.

## Files

- **`apps-wb-bc.jsx`** — two pane components:
  - `WBPane` — browser with nav strip (back/fwd/reload + URL bar
    with lock + SOP/JS/star pills) + bookmarks bar + page area +
    status strip (state + progress bar + cookie count).
  - `BCPane` — Cave manager split 60/40: caves table on the
    left (state badge + name + type pill + cap pills), detail
    panel on the right (cave glyph + KV rows + action hints +
    audit mini-strip).
- **`wave4-specs.jsx`** — spec sheet with concrete pixel widths,
  glyph mappings, status colors per state.

## Implementation

Ported to Rust as STUMP #133:

- **`src/ui/widgets.rs`** — extended with Wave-4 helpers:
  `host_swatch_color` (deterministic hash → 8-color palette),
  `draw_nav_btn` (28×28 icon button), `draw_bookmark_chip`
  (favicon swatch + host name), `draw_cave_glyph` (64×48 stylized
  container with dashed inner seal + corner notches + center
  node), `draw_action_hint` (`[shell] cmd  # comment` row).

- **`src/ui/apps/browser.rs`** — chrome surgically rewritten.
  Old tab bar / nav toolbar / progress bar / divider gated under
  `#[cfg(any())]` (compiled out, kept for diff context). New
  layout: nav strip (40px) + bookmarks bar (24px) + page area
  (existing render pipeline picks up between chrome strips) +
  status strip (24px) at the bottom. Lock icon reads
  `tls_pinning::current_mode()` to decide green/amber/red. SOP
  pill reads `origin::current_main_origin()` + `is_strict()`.
  JS pill reads `browser::js::is_enabled()`. Cookie count from
  `cookies::count()`. Loading-progress bar in the mid-segment
  of the status strip.

- **`src/ui/apps/caves_mgr.rs`** — full rewrite. Header strip
  ("CAVES" + slot count), 60/40 split body, footer with
  count + per-state mini-pills + nav hint. Selection state
  (`SELECTED_CAVE` static, arrow-key nav via `handle_key`,
  desktop dispatches `APP_CAVES` keystrokes the same way
  it does APP_FILES). Detail panel renders the selected cave
  with glyph + KV rows (NAME / STATE / TYPE / FS_KEY / CAPS /
  TOOLS / AUDIT / CREATED), action hints (`caves enter` cyan,
  `caves seal` amber for irreversible, `caves destroy`
  amber), and an audit mini-strip pinned to the bottom. Empty
  state shows the quickstart command list.

## Honest substitutions

| Design label | What we actually have |
|--------------|------------------------|
| WB bookmarks | static array of placeholder hostnames; not persisted |
| WB favicons | hash → 12×12 swatch color; no real favicon fetch |
| WB SOP host | reads main origin; truncated to first 24 chars |
| WB cookie count | global count from cookies::count (not per-host) |
| BC selection | SELECTED_CAVE index, arrow-key nav, no Enter action yet |
| BC FS_KEY | shows "c4e3d7a2" placeholder; real per-cave key derivation works but only the first 8 hex are safe to show |
| BC TOOLS | always "0 (kernel cave)" until we have docker-backed caves |
| BC AUDIT N | reads global audit ring; not per-cave filtered yet |
| BC CREATED | "0d ago" placeholder; we don't store creation timestamps |
| BC quickstart | inline shell commands, not buttons (keyboard-only OS) |

## Deviations

1. **Bookmarks aren't persisted.** Hardcoded list. Adding BatFS
   read/write per bookmark is a follow-up STUMP.
2. **No real favicon fetch.** Hash-derived swatch colors only.
3. **Star button is decorative.** No bookmark-add action wired.
4. **WB still uses the old page-rendering pipeline** for the
   middle area. The chrome around it is new; the rendered page
   itself looks the same as before.
5. **BC Enter does nothing yet.** Selection-only; future
   feature could open a per-cave detail view or attach.
